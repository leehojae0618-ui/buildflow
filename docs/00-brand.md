### 2. Brand Definition

BuildFlow는 사용자가 원하는 결과를 입력하면 실행 가능한 AI Workflow를 설계하는 AI Workflow Design Platform이다. 도구 자체보다 결과와 실행 경로를 먼저 제시한다. 추천, 설계, 비용 분석, 난이도 분석, 제한된 실행을 하나의 흐름으로 제공한다. 비개발자와 AI 입문자가 주요 사용자다.

### 3. Name Meaning

Build는 원하는 결과를 만든다는 뜻이다. Flow는 결과를 만들기 위한 단계와 연결 구조를 뜻한다. BuildFlow는 원하는 결과를 만드는 가장 적절한 흐름이라는 의미를 담는다. 이 이름은 사용자가 만들 수 있고, 순서가 있으며, 복잡성을 정리해주고, 실행으로 이어진다는 인상을 전달한다.

### 4. Brand Mission

AI와 자동화 경험이 없는 사용자도 원하는 결과를 만들 수 있도록 실행 가능한 Workflow를 설계한다.

### 5. Brand Vision

BuildFlow는 사용자가 AI Tool을 직접 조사하고 비교하는 시간을 줄이는 제품이 되어야 한다. 목표, 예산, 숙련도, 사용 환경에 맞는 실행 구조를 제공해야 한다. AI Tool이 바뀌어도 결과 중심 구조는 유지되어야 한다. 장기적으로 Workflow 지식베이스와 운영 관리 기능을 핵심 자산으로 만든다.

### 6. Core Product Philosophy

1. Tool보다 Result: 사용자는 도구 목록보다 원하는 결과를 먼저 본다.
2. Recommendation보다 Design: 추천은 시작점이고, 실제 가치는 설계에 있다.
3. Design보다 Executability: 보기 좋은 설계보다 실행 가능한 구조가 중요하다.
4. 복잡성은 시스템이 처리: 사용자가 복잡한 연결을 직접 해석하지 않게 한다.
5. 사용자는 선택과 확인에 집중: 시스템이 구조를 만들고 사용자는 결정한다.
6. 빠른 설계 후 점진적 실행: 처음부터 완성형 자동화를 강요하지 않는다.
7. 특정 AI Provider에 종속되지 않음: 결과 중심 구조를 유지한다.
8. 1인 개발이 유지할 수 있는 구조: 범위를 통제 가능한 수준으로 유지한다.

### 7. Primary Message

메인 메시지는 `무엇을 만들고 싶나요?`이다. 이 문장은 사용자의 목표를 먼저 묻고, 도구보다 결과를 우선하는 제품 방향을 보여준다. 보조 설명 문구는 `원하는 결과를 입력하면 BuildFlow가 필요한 AI 도구, Workflow, 비용, 난이도와 실행 방법을 설계합니다.`이다.

### 8. Slogan System

Primary Slogan은 `Build the result, not the tools.`이다. 한국어 의미는 `도구가 아니라 결과를 만드세요.`이다.

Secondary Slogan 후보는 다음과 같다.

- From idea to workflow.
- Build what matters.
- Design. Build. Automate.
- Your idea, built into a workflow.

Primary와 Secondary를 구분하며, 슬로건은 임의로 추가하지 않는다.

### 9. Positioning

BuildFlow는 `AI Workflow Design Platform`이자 `AI Result Platform`이다.

| 구분 | 기존 AI Tool 서비스 | BuildFlow |
| --- | --- | --- |
| 시작점 | Tool 검색 | 원하는 결과 |
| 사용자 행동 | 비교와 학습 | 목표 입력 |
| 핵심 출력 | Tool 목록 | 실행 가능한 Workflow |
| 비용 | 사용자가 직접 계산 | 전체 예상 비용 제공 |
| 난이도 | 사용자가 판단 | 시스템이 분석 |
| 실행 | 외부에서 직접 구성 | 제한 실행 또는 연결 안내 |

### 10. What BuildFlow Is Not

BuildFlow는 다음이 아니다.

- AI Tool Directory: 도구 목록만 제공하는 서비스가 아니기 때문이다.
- ChatGPT Wrapper: 대화 UI만 감싼 제품이 아니기 때문이다.
- 범용 AI Chat 서비스: 일반 채팅 기능이 핵심이 아니기 때문이다.
- n8n 대체 서비스: 실행 엔진 자체를 대체하려는 제품이 아니기 때문이다.
- 범용 노코드 Workflow Builder: 모든 워크플로를 자유 편집하는 빌더가 아니기 때문이다.
- 개발자 IDE: 코드를 작성하는 개발 환경이 아니기 때문이다.
- AI Agent Framework: 에이전트 프레임워크를 제공하는 제품이 아니기 때문이다.
- API 재판매 서비스: API를 다시 판매하는 사업이 아니기 때문이다.
- 모든 자동화를 직접 실행하는 플랫폼: 지원 가능한 범위를 넘는 전면 실행 서비스가 아니기 때문이다.

### 11. Target Audience

Primary Audience:

- AI 입문자
- 1인 사업자
- 쇼핑몰 운영자
- 마케터
- 유튜버 및 콘텐츠 제작자
- 반복 업무가 많은 직장인
- 비개발자

Initial Non-Target:

- 전문 개발자
- 대규모 엔터프라이즈
- 복잡한 멀티에이전트 시스템을 만드는 팀
- 자체 인프라 운영을 원하는 조직

비개발자를 우선하는 이유는 BuildFlow의 핵심 가치가 도구 학습이 아니라 결과 중심 설계에 있기 때문이다. 비개발자는 Tool 비교보다 목표를 빠르게 구조화해주는 안내가 더 필요하다.

### 12. Brand Personality

- Clear: 무엇을 제공하는지 바로 이해할 수 있어야 한다.
- Practical: 실제 실행과 연결되는 설명을 우선한다.
- Reliable: 과장보다 정확성을 우선한다.
- Calm: 복잡한 문제를 차분하게 정리해준다.
- Intelligent: 조건과 맥락을 구조적으로 반영한다.
- Non-technical: 기술 지식이 없어도 이해할 수 있다.
- Result-oriented: 항상 결과와 실행을 먼저 생각한다.

BuildFlow는 과장되거나 장난스러운 브랜드가 아니다. 지나치게 개발자 중심적이거나 난해한 표현도 피한다.

### 13. Tone of Voice

#### 사용해야 하는 표현

- 결과
- 목표
- 단계
- 비용
- 난이도
- 준비사항
- 연결
- 실행
- 관리
- 추천 이유

#### 피해야 하는 표현

- 혁신적인
- 압도적인
- 마법 같은
- 모든 것을 자동화
- 클릭 한 번으로 완성
- 무조건 성공
- 완벽한 AI
- 세계 최초

#### 문장 원칙

- 짧고 명확하게 작성한다.
- 전문용어는 설명 없이 사용하지 않는다.
- 사용자가 해야 할 행동을 명확히 안내한다.
- 불가능한 것은 가능하다고 표현하지 않는다.
- 비용과 제한을 숨기지 않는다.

#### 사용 예시

- 목표를 입력하면 적합한 Workflow를 보여드립니다.
- 예상 비용과 난이도를 함께 확인하세요.
- 필요한 계정이 있으면 먼저 안내해드립니다.
- 제한된 실행은 현재 지원 범위 안에서만 가능합니다.
- 수정 요청을 통해 Workflow를 다시 설계할 수 있습니다.

#### 잘못된 예시

- 혁신적인 자동화 경험을 제공합니다.
- 클릭 한 번으로 모든 것이 완성됩니다.
- 무조건 성공하는 AI 솔루션입니다.
- 마법처럼 업무를 해결합니다.
- 세계 최초의 완벽한 Workflow 플랫폼입니다.

### 14. Naming System

| English | Korean UI | Definition |
| --- | --- | --- |
| Project | 프로젝트 | 사용자가 달성하려는 하나의 목표 단위 |
| Goal | 목표 | 사용자가 만들고 싶은 결과 |
| Build | 빌드 | 목표를 Workflow 설계로 변환하는 과정 |
| Recommendation | 추천 | 조건에 맞는 Workflow 후보 |
| Workflow | 워크플로 | 결과를 만들기 위한 단계와 Tool 연결 구조 |
| Step | 단계 | Workflow를 구성하는 개별 작업 |
| Tool | 도구 | Workflow에서 사용하는 AI 또는 외부 서비스 |
| Integration | 연동 | 외부 계정 또는 API 연결 |
| Execution | 실행 | 설계된 일부 작업을 실제로 수행하는 과정 |
| Usage | 사용량 | 추천, 생성, 실행 등의 소비 기록 |
| Template | 템플릿 | 재사용 가능한 검증된 Workflow 구조 |

사용자 UI에서는 필요 이상으로 영어를 노출하지 않는다. 다만 `Workflow`, `API`, `n8n`처럼 업계에서 통용되는 용어는 설명과 함께 사용할 수 있다.

### 15. Feature Naming

핵심 기능 이름은 아래를 임시 기준으로 사용한다.

- New Project → 새 프로젝트
- Goal Input → 목표 입력
- Build → 설계 생성
- Recommendation → 추천 결과
- Workflow Detail → 워크플로 상세
- Cost Estimate → 예상 비용
- Difficulty → 난이도
- Required Setup → 필요한 준비
- Test Run → 테스트 실행
- My Projects → 내 프로젝트
- Usage → 사용량
- Settings → 설정

`Build`는 핵심 기능명으로 사용할 수 있으나 사용자에게 의미가 불분명한 화면에서는 `설계 생성`으로 표기한다. `Flow Engine`은 내부 추천 엔진의 가칭이며 외부 브랜드로 확정하지 않는다.

### 16. Visual Direction

초기 방향은 Minimal, Structured, Professional, Calm, High readability, Modern SaaS다. 피해야 할 방향은 과도한 네온, 복잡한 그라데이션, 지나친 사이버펑크, AI 로봇 이미지, 빛나는 뇌 이미지, Tool 로고 과다 노출, 장식 중심 UI, 지나치게 어두워 가독성이 낮은 UI다. 초기 색상 방향은 Black, White, Neutral Gray, One primary accent color이며 색상 Hex 값은 아직 `TBD`다.

### 17. Logo Direction

아직 실제 로고를 제작하지 않는다. 초기 로고는 Wordmark 중심으로 하고, `BuildFlow`의 B와 F는 대문자로 사용한다. 작은 화면에서도 읽을 수 있어야 하며, 복잡한 심볼보다 텍스트 인지성을 우선한다. AI 로봇, 뇌, 번개 아이콘은 사용하지 않는다. 향후 Flow 또는 연결 구조를 추상화한 심볼은 가능하다.

### 20. Brand Guardrails

- BuildFlow를 단순 Tool 검색 서비스로 설명하지 않는다.
- 사용자가 모든 API와 자동화를 이해한다고 가정하지 않는다.
- 실행할 수 없는 기능을 실행 가능하다고 표시하지 않는다.
- 특정 Provider를 무조건 최고라고 표현하지 않는다.
- Tool 수를 핵심 경쟁력으로 강조하지 않는다.
- Workflow 품질과 실행 가능성을 우선한다.
- 추천 이유와 제한을 함께 표시한다.
- 사용자 API 비용 부담 원칙을 투명하게 안내한다.
- 디자인보다 정보 구조와 이해 가능성을 우선한다.
